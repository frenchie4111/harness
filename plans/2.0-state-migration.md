# 2.0 — Main-process state migration

This is the historical record of the 2.0 architecture refactor and the
forward-looking plan for what comes next. The architecture itself is
documented in [CLAUDE.md](../CLAUDE.md) — read that first if you're
trying to understand how the app works today. This doc is the **why**
and the **what's next**.

## Why we did this

Through 1.x, Harness kept most of its control logic in the React
renderer: worktree lists, PR polling cadence, pending-worktree creation
state machines, hooks consent flow, pane/tab model, terminal status
caches, activity bookkeeping. The main process was a passive collection
of IPC handlers that the renderer orchestrated.

This blocked four directions we wanted to go:

1. **MCP lifecycle** — MCP servers live naturally in Node but were
   driven from the renderer.
2. **Headless / background mode** — couldn't run without a window
   because state lived in the renderer.
3. **Alternate frontends (web/mobile)** — a network client would have
   to reimplement the entire state layer from scratch.
4. **Multi-window workspaces** — needs shared state to be portable
   across windows.

The migration moved all non-UI state into the main process. After 2.0,
adding any of these four features is a layer on top of a stable
foundation rather than a rewrite.

## What 2.0 actually shipped

**No user-visible changes.** This is an internal refactor. Every flow
you use today still works the same way; the implementation just moved.

### State slices

All shared world state now lives in one of eight slices under
`src/shared/state/`. Each slice has a typed state interface, a
discriminated event union, a pure reducer, and unit tests.

| Slice | What's in it |
|---|---|
| `settings` | theme, hotkeys, claudeCommand, fonts, editor, env vars, worktreeBase, mergeStrategy, hasGithubToken, etc. (13 fields) |
| `prs` | byPath PRStatus, mergedByPath, loading flag |
| `worktrees` | flat list, repoRoots, pending creation FSM entries |
| `terminals` | per-tab statuses, pendingTools, shellActivity, panes (the full pane/tab tree), lastActive timestamps |
| `onboarding` | quest step |
| `hooks` | consent + justInstalled |
| `updater` | auto-updater status |
| `repoConfigs` | byRepo: per-repo `.harness.json` contents |

The renderer reads slices via hooks (`useSettings()`, `usePrs()`,
`usePanes()`, etc.) backed by `useSyncExternalStore`. Both main and
the renderer apply the **same** reducer code from `src/shared/state/`,
which is what keeps them in sync without any custom diffing.

See CLAUDE.md "Architecture (read this before touching state)" for
the full pattern, the end-to-end mutation flow, and the 5-file
checklist for adding a new piece of shared state.

### Notable scope changes from the original plan

- **Workspace layer (slice 6) deferred to 2.1.** The original plan
  bundled the workspace UI feature with the state migration. Splitting
  them gives 2.0 a coherent "internal refactor" identity and means the
  workspace feature can be designed with care rather than rushed.
- **"Binary terminal stream" was a misconception.** An earlier draft
  treated terminal data as needing a special transport. It doesn't —
  it's just a side-effect signal that uses its own IPC channel, which
  is exactly the right shape. A future WebSocket transport can carry
  state events + terminal data on one connection with a `kind`
  discriminator. No special abstraction needed.
- **Activity migration was deferred from slice 3 to slice 5.** The
  `lastRecordedActivity` dedup needed terminal statuses to be in the
  store first.
- **Panes were almost left renderer-owned, then weren't.** Mid-slice-5
  the user pointed out that "renderer just renders" was the actual
  goal of the migration, not "minimum viable for slice 6." Slice 5 was
  expanded to include the full pane/tab tree as a result.

### What's still in the renderer (and why)

After the migration, App.tsx has ~20 `useState`/`useRef` declarations.
Every one is per-client UI state — focus, modal visibility, sidebar
widths, tree expansion, per-session bookkeeping. None of it would
ever be shared across clients even with infinite engineering time.
This is the irreducible per-viewer state every frontend has to own.

A future web/mobile client would implement these same per-client
values and read everything else from the store snapshot.

## Future work

### 2.1+ — Workspace layer

The big remaining feature: per-workspace process model with isolated
state, MCP, and config. After 2.0 ships this is mostly UX/process work
on a stable foundation.

#### Process model

- **One Electron process per workspace.** Each window is its own
  process, launched with `--workspace <id>`. Zero cross-process state
  sharing beyond a few coordination files.
- **No broker process.** Each workspace binds its own MCP endpoint at
  a unix domain socket inside its workspace dir. A Claude running in
  workspace A can only see workspace A's worktrees — correct isolation
  boundary, falls out for free.
- **Workspace = `{id, name, repos[], windowBounds, createdAt}`**,
  persisted under `userData/workspaces/<id>/`.

#### Directory layout

```
userData/
├── global.json                    # theme, claude command, hotkeys,
│                                  # github token, editor, font, etc.
├── workspaces.json                # registry: [{id, name}, ...]
├── updater.lock                   # first-acquirer runs the updater
└── workspaces/
    └── <id>/
        ├── config.json            # panes, worktree scripts, onboarding
        ├── activity.json          # activity log scoped to this workspace
        ├── control.sock           # harness-control MCP endpoint
        └── terminal-history/      # xterm scrollback dumps
```

Global settings are owned by whichever process writes last; other
processes pick up changes via `fs.watch('global.json')`. ~50 lines.

#### Persistence migration

Existing users' single `config.json` splits into `global.json` +
`workspaces/<default-id>/config.json`. The default workspace inherits
their current `repoRoots`. Add the migration to
`persistence-migrations.ts` + tests. **Risk area** — if this migration
goes wrong, people lose data.

#### Boot flow

`app.whenReady()` parses `--workspace <id>` arg:
- No arg → show workspace picker (or open most-recent)
- With id → load that workspace's config and run

#### MCP per workspace

`control-server.ts` binds a unix socket at
`workspaces/<id>/control.sock` instead of TCP. Each workspace's Claude
tabs get an MCP config pointing at their own socket.
`mcp-config.ts` writes the per-workspace path. **Risk area** —
existing Claude sessions need to reconnect after the upgrade.

#### Coordination files

- `userData/updater.lock` — first acquirer runs the auto-updater
- Remove `app.requestSingleInstanceLock()` (verify it's even set first)

#### UI

- "New Workspace…" menu item → modal with name + repo picker → spawns
  process
- "Workspaces" menu listing known workspaces, click to focus or spawn
- Workspace picker on cold boot when no `--workspace` arg
- Open-from-Finder behavior: opens the picker (safest default)

#### Rough scope

6-10 commits, ~1200 lines net. The migration prerequisite is done so
this is process-model + UX + persistence migration only.

### When there's a second client — WebSocket transport

The state event stream and method dispatch are already in a
transport-agnostic shape. A future WebSocket transport would:

- Run inside `control-server.ts` (already an HTTP server) on a `/ws`
  upgrade route
- On connect, send snapshot, subscribe to event bus, accept JSON-RPC
  method calls
- **Multiplex everything on one connection** with a `kind` discriminator:

  ```ts
  type WireMessage =
    | { kind: 'state'; event: StateEvent; seq: number }
    | { kind: 'terminal-data'; terminalId: string; data: string }
    | { kind: 'terminal-exit'; terminalId: string; exitCode: number }
    | { kind: 'script-event'; runId: string; ... }
  ```

- Use the existing `seq` field for reconnect/replay: client sends its
  last-seen seq, server replays missed events from a short ring buffer
  or sends a fresh snapshot if the gap is too big

The "binary frames" thing isn't a real concern — terminal output is
mostly ASCII printable text + escape sequences, and JSON encoding is
~1.1× overhead at the rates terminals actually generate. WebSocket
binary frames are available if needed but text frames work.

This is the second user-visible feature 2.0 enables. Best designed
when there's a concrete second client (web/mobile) to test against,
not in a vacuum.

### Smaller follow-ups (any time)

- **Invert the `PanesFSM` ⟷ `WorktreesFSM` coupling.** Right now
  `WorktreesFSM`'s `onWorktreeCreated` callback closes over `panesFSM`,
  and `main/index.ts`'s construction order is load-bearing (with a
  warning comment block). Cleaner: have `PanesFSM` subscribe to
  `worktrees/listChanged` itself and call `ensureInitialized` on any
  unowned worktree. Inverts the dependency direction and removes the
  construction-order requirement.
- **Promote `installHooksForAcceptedWorktrees`** from an inline
  function in `main/index.ts` to a `HooksInstaller` class for
  consistency with `PRPoller`/`WorktreesFSM`/`PanesFSM`/`ActivityDeriver`.
- **App.tsx sub-component extraction.** 815 lines is manageable but
  could be smaller. The JSX cluster is the biggest remaining chunk.
  Pure code organization, not blocking anything.
- **`main/index.ts` is 1310 lines.** The slice migrations relocated
  orchestration code there. Slice 6 will likely break it up further
  (per-workspace process means simpler boot). Not worth refactoring
  before slice 6.
